# Релиз и откат (Portainer)

## Окружения

Рекомендуется два стека:

- **staging** — тот же образ/ветка, отдельные `DB_*` или префикс БД; прогон `verify-schema` + `smoke-e2e.ps1`.
- **production** — боевые секреты и БД; обновления только после успешного staging.

## Стандартный релиз

1. Изменения в `main` (или релизной ветке) → `git push`.
2. В Portainer: **Pull and redeploy** стека (или пересборка image из Git).
3. Убедиться, что контейнер стартовал, миграции применились при старте (`initializeDatabase` / логи).
4. Проверить `GET /health`.
5. По желанию: `smoke-e2e.ps1` против прод-URL.

Миграции вручную (если нужно без рестарта):

```bash
docker exec -it creative_collective_api sh -c "npm run migrate"
```

## Миграции: правила

- Новые SQL только в `backend/migrations_uuid/` с понятным префиксом и идемпотентностью (`IF NOT EXISTS` / hotfix через `information_schema`).
- Локально проверить `npm run migrate` на копии прод-данных.
- Не редактировать уже применённые файлы миграций на проде — только новые файлы.

## Откат

1. **Код**: в Portainer указать предыдущий Git commit / image tag → redeploy.
2. **БД**: откат миграций автоматически не делается. План:
   - восстановить БД из бэкапа **до** релиза, или
   - подготовить forward-fix миграцию.
3. После отката снова проверить `/health` и smoke.

## Чеклист перед прод-релизом

- [ ] Staging прошёл smoke.
- [ ] Бэкап MySQL снят (или окно обслуживания согласовано).
- [ ] Секреты не в репозитории.
- [ ] `CORS_ORIGIN` и rate limit соответствуют проду.
